# HU-10: Cierre y Reapertura de Operaciones con Validación de Pagos

**Como** empleado de la concesionaria,
**quiero** que el sistema valide los pagos, cierre automáticamente la operación al saldarse y bloquee acciones incompatibles con el estado de la operación,
**para** evitar sobrepagos, inconsistencias de estado y ediciones accidentales en operaciones cerradas.

---

## Descripción

Actualmente el sistema permite registrar pagos que superan el total de la operación y no cambia el estado al cerrarse. Esta historia incorpora tres mejoras relacionadas:

1. **Validación de sobrepago:** el sistema rechaza cualquier pago cuyo monto supere el saldo pendiente real.
2. **Cierre automático:** al registrar el pago que deja el saldo en $0, la operación cambia automáticamente su estado a `cerrada` y se muestra un modal de confirmación.
3. **Bloqueo de acciones según estado:** cuando una operación está cerrada, se deshabilita la carga de nuevos pagos y la edición de campos; si el empleado quiere editar, debe confirmar la reapertura de la operación explícitamente. La carga de gastos permanece habilitada en todo momento.

---

## Criterios de aceptación

- [ ] El sistema rechaza un pago cuyo monto supere el saldo pendiente real, mostrando el mensaje: *"El monto supera el saldo pendiente"*
- [ ] El botón "Pagó todo" en el modal de pago autocompleta con el saldo pendiente real (precio total − suma de pagos ya registrados), no con el precio total bruto
- [ ] Al registrar el pago que salda la operación, el estado cambia automáticamente a `cerrada` en la base de datos
- [ ] Tras el cierre automático, se muestra un modal "Operación cerrada" con un único botón "Aceptar" que redirige a la tabla de operaciones
- [ ] En una operación cerrada, el botón "Agregar pago" está deshabilitado y muestra un hint informativo al usuario
- [ ] En una operación cerrada (reabierta), el botón "Cerrar operación" está disponible y al hacer clic muestra un modal de advertencia: *"¿Estás seguro de cerrar la operación?"*
- [ ] Si el empleado confirma el cierre desde ese modal, el estado cambia a `cerrada` directamente, sin solicitar pago
- [ ] En la vista de edición, si la operación está cerrada, todos los campos del formulario están deshabilitados y se muestra un banner: *"Esta operación está cerrada. Para editarla, primero debés reabrirla."*
- [ ] Al hacer clic en el botón "Editar" de una operación cerrada, aparece un modal de advertencia: *"¿Deseás reabrir la operación?"*
- [ ] Si el empleado confirma la reapertura, el estado cambia a `abierta` y se accede al modo edición normalmente
- [ ] Si el empleado cancela la reapertura, no ocurre ningún cambio
- [ ] La sección de gastos (`OperationExpensesSection`) permanece habilitada independientemente del estado de la operación

---

## Flujos

### Flujo principal — Cierre automático al saldar la operación

1. El empleado abre una operación y registra un pago desde "Cobranzas" o desde el botón "Cerrar operación".
2. El sistema valida que el monto no supere el saldo pendiente real.
3. El pago se registra y el saldo queda en $0.
4. El sistema cambia automáticamente el estado de la operación a `cerrada`.
5. Aparece un modal "Operación cerrada" con un botón "Aceptar".
6. El empleado hace clic en "Aceptar" y es redirigido a la tabla de operaciones.

### Flujo alternativo 1 — Sobrepago

1. El empleado ingresa un monto que supera el saldo pendiente real.
2. El sistema rechaza el pago y muestra el error: *"El monto supera el saldo pendiente"*.
3. El empleado corrige el monto y reintenta.

### Flujo alternativo 2 — Intentar pagar en operación cerrada (reabierta)

1. El empleado abre una operación que fue cerrada y luego reabierta.
2. El botón "Agregar pago" está deshabilitado; se muestra un hint informativo indicando que la operación ya fue saldada.
3. El botón "Cerrar operación" está disponible.
4. El empleado hace clic en "Cerrar operación"; aparece un modal de advertencia: *"¿Estás seguro de cerrar la operación?"*
5. Si confirma → el estado cambia directamente a `cerrada` sin requerir un nuevo pago.
6. Si cancela → no ocurre ningún cambio.

### Flujo alternativo 3 — Intentar editar una operación cerrada

1. El empleado hace clic en "Editar" de una operación con estado `cerrada`.
2. Aparece un modal de advertencia: *"¿Deseás reabrir la operación?"*
3. Si confirma → el sistema cambia el estado a `abierta` y entra al modo edición normalmente.
4. Si cancela → no ocurre ningún cambio; la operación sigue cerrada.

### Flujo alternativo 4 — "Pagó todo" con saldo real

1. El empleado abre el modal de pago desde "Cerrar operación".
2. Hace clic en el botón "Pagó todo".
3. El campo monto se autocompleta con el saldo pendiente real (precio total − suma de pagos ya registrados).

---

## Notas técnicas

⚠️ **Base de datos:** El campo `estado` de la operación ya existe. El cambio a `cerrada` se aplica automáticamente desde el endpoint de pagos cuando el saldo resultante es $0. La reapertura actualiza ese mismo campo a `abierta` desde un endpoint dedicado o desde el endpoint de edición.
