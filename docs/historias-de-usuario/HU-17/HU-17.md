# HU-17: Optimización de almacenamiento y carga de fotos de vehículos

**Como** usuario interno de la concesionaria (vendedor o administrador),
**quiero** que las fotos que subo sean comprimidas automáticamente y que en el detalle del vehículo los datos se vean de inmediato sin esperar a que carguen todas las imágenes,
**para** reducir el espacio ocupado en la base de datos y mejorar la experiencia de navegación del sistema.

## Descripción

Actualmente las fotos de vehículos se guardan en la base de datos tal como el usuario las sube, sin ningún tipo de compresión ni redimensionado. Esto genera registros muy pesados que consumen rápidamente el almacenamiento disponible. Además, en la vista de detalle de un vehículo, todas las fotos intentan cargarse al mismo tiempo, lo que puede hacer que el usuario tenga que esperar para ver información básica como precio, marca o año.

Esta historia busca resolver ambos problemas: comprimir y redimensionar las fotos en el momento de la subida (generando una versión thumbnail y una versión full), y aplicar lazy loading en las fotos secundarias del detalle para que los datos textuales sean visibles de inmediato.

## Criterios de aceptación

- [ ] Al subir una foto, el sistema valida que tenga al menos 800px en su lado más largo antes de procesarla.
- [ ] Si una foto no cumple el mínimo de resolución, se rechaza con un mensaje de error claro que indica el requisito mínimo. Las demás fotos válidas del mismo lote se procesan normalmente.
- [ ] Cada foto válida se convierte automáticamente a formato WebP al momento de la subida.
- [ ] Se generan dos versiones de cada foto: thumbnail (400px en el lado más largo) para uso en listados, y full (1280px en el lado más largo) para uso en el detalle.
- [ ] Las imágenes no se agrandan si su resolución original es menor a los límites definidos (no se hace upscaling).
- [ ] En el listado de stock, las miniaturas de los vehículos se sirven usando la versión thumbnail.
- [ ] En el detalle del vehículo, los datos textuales (marca, modelo, año, precio, etc.) son visibles de inmediato, sin esperar a que carguen las fotos.
- [ ] La foto principal del detalle carga en versión full con un placeholder visible mientras se descarga.
- [ ] Las fotos secundarias en la tira de miniaturas del detalle usan lazy loading: se cargan solo cuando el navegador las necesita.
- [ ] Si un vehículo no tiene fotos, el sistema muestra un ícono o placeholder genérico tanto en el listado como en el detalle.

## Flujos

### Flujo principal — Subida de foto

1. El usuario carga una o más fotos al crear o editar un vehículo.
2. Por cada foto, el sistema verifica que el lado más largo tenga al menos 800px.
3. El sistema procesa cada foto válida con Sharp: la convierte a WebP y genera dos versiones — thumbnail (400px) y full (1280px), manteniendo la proporción original.
4. Ambas versiones se guardan en la base de datos asociadas al vehículo.
5. La operación se completa y el usuario ve las fotos disponibles.

### Flujo principal — Listado de stock

1. El usuario navega al listado de stock.
2. El sistema devuelve los datos de cada vehículo junto con el ID de la primera foto.
3. El listado muestra la versión thumbnail de la primera foto de cada vehículo.
4. La carga es significativamente más rápida que con las fotos originales.

### Flujo principal — Detalle de vehículo

1. El usuario abre el detalle de un vehículo.
2. Los datos textuales del vehículo (marca, modelo, año, precio, descripción, etc.) se muestran de inmediato al renderizar la página.
3. La foto principal comienza a cargarse en versión full; mientras tanto se muestra un placeholder.
4. Una vez cargada, la foto principal reemplaza el placeholder.
5. Las fotos secundarias en la tira inferior usan lazy loading: el navegador las solicita a medida que son necesarias, sin bloquear la visualización de los datos.

### Flujo alternativo — Foto con resolución insuficiente

1. El usuario intenta subir una foto cuyo lado más largo mide menos de 800px.
2. El sistema rechaza esa foto y muestra un mensaje de error: _"La imagen debe tener al menos 800px en su lado más largo."_
3. Si se subieron varias fotos en el mismo lote, las demás imágenes válidas se procesan normalmente.
4. El usuario puede seleccionar otra imagen de mayor resolución y volver a intentarlo.

### Flujo alternativo — Sin fotos

1. El vehículo no tiene ninguna foto asociada.
2. En el listado, se muestra un ícono genérico de auto en lugar de una imagen.
3. En el detalle, el espacio de la foto principal muestra el mismo placeholder genérico.

## Notas técnicas

⚠️ **Base de datos:** Esta historia requiere modificar la tabla `VehiclePhoto` para almacenar dos versiones de cada imagen: una versión thumbnail (`datosThumb`, Bytes) y una versión full (`datos`, Bytes, que pasa a guardar la imagen comprimida en lugar del original). También se requiere agregar un campo `mimeType` con valor `"image/webp"` para todas las fotos nuevas. Se necesita una migración de Prisma. Las fotos ya existentes en la base de datos no se migran automáticamente y seguirán sirviéndose desde el campo `datos` original.

⚠️ **Dependencia externa:** Esta historia requiere instalar la librería **Sharp** (`npm install sharp`) en el proyecto Next.js para el procesamiento de imágenes en el servidor.

⚠️ **Endpoints afectados:** Los endpoints de subida de fotos en `/api/stock` y `/api/operations` deben incorporar el procesamiento con Sharp. El endpoint de servicio de fotos `/api/stock/[id]/photos/[photoId]` debe aceptar un parámetro opcional `?thumb=true` para devolver la versión thumbnail en lugar de la full.
