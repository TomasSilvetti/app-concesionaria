# HU-15: Suite de tests de integración para API routes críticas

**Como** equipo de desarrollo,
**quiero** tener tests de integración que ejecuten las API routes contra una base de datos PostgreSQL real,
**para** detectar regresiones en los flujos críticos del sistema (operaciones, pagos, gastos, stock, usuarios) antes de que lleguen a producción, cubriendo comportamientos que los unit tests no pueden verificar.

## Descripción

HU-14 estableció la infraestructura de testing y los primeros unit tests sobre funciones puras. Sin embargo, la lógica más crítica del sistema vive en las API routes: creación de operaciones, registro de pagos que recalculan márgenes, gastos que actualizan ingresos netos, y el aislamiento multi-tenant que impide que un cliente acceda a datos de otro. Ninguna de estas cosas puede verificarse con unit tests sin mockear tanto que el test pierde valor.

Esta historia cubre la creación de tests de integración que levantan requests reales contra las API routes, usando una base de datos PostgreSQL de test dedicada (local en Docker, y en CI como service container). Cada suite siembra sus propios datos y los limpia antes de cada test.

## Criterios de aceptación

- [ ] Existe un archivo `vitest.config.integration.ts` separado del config de unit tests
- [ ] Existe un script `npm run test:integration` que corre solo los tests de integración
- [ ] Existe una función utilitaria `src/__tests__/integration/helpers/seed.ts` para crear datos de prueba reutilizables (cliente, usuario, operación, vehículo, etc.)
- [ ] Existe una función utilitaria `src/__tests__/integration/helpers/auth.ts` para generar tokens JWT de test sin pasar por el flujo de login
- [ ] Cada suite usa `beforeEach` para limpiar y resembrar los datos que necesita
- [ ] El workflow `.github/workflows/tests.yml` incluye un service container de PostgreSQL y corre los tests de integración además de los unit tests
- [ ] Si algún test de integración falla, el workflow bloquea el merge
- [ ] Los tests de integración no dependen de datos preexistentes ni de orden de ejecución

### Operaciones (Nivel 1)
- [ ] POST `/api/operations` crea una operación y devuelve 201 con los datos correctos
- [ ] POST `/api/operations` con `tipoOperacion` inválido devuelve 400
- [ ] POST `/api/operations` sin sesión activa devuelve 401
- [ ] GET `/api/operations` devuelve solo las operaciones del `clienteId` del usuario autenticado
- [ ] PATCH `/api/operations/[id]` al actualizar `precioVentaTotal` recalcula `ingresosNetos` y `comision` correctamente en la BD
- [ ] PATCH `/api/operations/[id]` con un `id` que no pertenece al cliente devuelve 403 o 404

### Pagos (Nivel 1)
- [ ] POST `/api/operations/[id]/pagos` registra el pago y actualiza el saldo pendiente de la operación
- [ ] POST `/api/operations/[id]/pagos` con monto negativo devuelve 400
- [ ] GET `/api/operations/[id]/pagos` devuelve solo los pagos de esa operación

### Gastos asociados (Nivel 1)
- [ ] POST `/api/operations/[id]/expenses` agrega el gasto y recalcula `gastosAsociados` e `ingresosNetos` en la operación
- [ ] DELETE `/api/operations/[id]/expenses/[gastoId]` elimina el gasto y revierte el recálculo en la operación

### Stock (Nivel 2)
- [ ] POST `/api/stock` crea un vehículo asociado al `clienteId` correcto
- [ ] DELETE `/api/stock/[id]` devuelve 409 si el vehículo está vinculado a una operación activa
- [ ] GET `/api/stock/disponibles` devuelve solo vehículos con estado `disponible`

### Usuarios y multi-tenancy (Nivel 2)
- [ ] POST `/api/users` con `rol=usuario` autenticado devuelve 403
- [ ] POST `/api/users` con `rol=admin` autenticado crea el usuario correctamente
- [ ] PATCH `/api/users/[id]` puede activar y desactivar un usuario
- [ ] GET `/api/clients` con `rol=usuario` devuelve solo el cliente propio, no los de otros

## Flujos

### Flujo principal (escenario de éxito)

1. El developer corre `npm run test:integration` localmente (requiere Docker con PostgreSQL corriendo)
2. Vitest descubre todos los archivos `*.integration.test.ts`
3. Antes de cada test, `beforeEach` limpia las tablas relevantes y siembra los datos necesarios
4. El test ejecuta un fetch real contra la Next.js API route (usando `testClient` o similar)
5. Se verifican: status HTTP, body de respuesta, y estado de la BD post-request
6. Todos los tests pasan y la suite queda en verde
7. En CI, GitHub Actions levanta el service container de PostgreSQL, corre migraciones, y ejecuta la misma suite

### Flujo alternativo 1 — BD de test no disponible

1. El developer intenta correr `npm run test:integration` sin Docker corriendo
2. Prisma no puede conectarse a `DATABASE_TEST_URL`
3. La suite falla con un error de conexión claro (no un timeout silencioso)
4. El developer levanta Docker y reintenta

### Flujo alternativo 2 — Test deja datos sucios por crash

1. Un test crashea antes de que `beforeEach` del siguiente test limpie la BD
2. El siguiente test encuentra datos inesperados
3. Como cada `beforeEach` hace un delete explícito antes de sembrar, los datos sucios son eliminados antes de que el test corra
4. El test pasa correctamente

### Flujo alternativo 3 — Violación de aislamiento multi-tenant detectada

1. Un test verifica que el usuario del cliente A no puede ver datos del cliente B
2. El test crea datos para dos clientes distintos y autentica como cliente A
3. La API devuelve datos del cliente A solamente
4. Si por un bug devolviera datos del cliente B, el test falla y bloquea el merge

## Notas técnicas

- **Framework:** Vitest (ya instalado) con config separada para integración
- **BD de test local:** PostgreSQL en Docker, base de datos `nordem_test` (separada de `nordem_dev`)
- **BD de test en CI:** GitHub Actions service container `postgres:15`
- **Variable de entorno:** `DATABASE_TEST_URL` apunta a la BD de test; los tests de integración deben usar esta variable, no `DATABASE_URL`
- **Autenticación en tests:** generar JWT firmados con `NEXTAUTH_SECRET` directamente, sin pasar por el endpoint de login
- **Limpieza:** `beforeEach` con deletes en orden correcto (respetar foreign keys); no usar `prisma migrate reset` por ser demasiado lento
- **Archivos a crear:**
  - `vitest.config.integration.ts`
  - `src/__tests__/integration/helpers/seed.ts`
  - `src/__tests__/integration/helpers/auth.ts`
  - `src/__tests__/integration/helpers/db.ts` — cliente Prisma apuntando a BD de test
  - `src/__tests__/integration/operations.integration.test.ts`
  - `src/__tests__/integration/pagos.integration.test.ts`
  - `src/__tests__/integration/expenses.integration.test.ts`
  - `src/__tests__/integration/stock.integration.test.ts`
  - `src/__tests__/integration/users.integration.test.ts`
- **Archivos a modificar:**
  - `.github/workflows/tests.yml` — agregar service container de PostgreSQL, correr migraciones, y ejecutar `npm run test:integration`
  - `package.json` — agregar script `test:integration`
- **Prerequisito:** HU-14 (Vitest configurado, CI existente)
- **No incluye:** tests E2E con browser, tests de componentes React, tests de uploads de archivos/fotos
