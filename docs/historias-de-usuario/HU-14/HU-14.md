# HU-14: Suite de unit tests automatizada con bloqueo en CI

**Como** equipo de desarrollo,
**quiero** tener una suite de unit tests que se ejecute automáticamente en cada push a `main`,
**para** detectar regresiones en la lógica crítica del sistema antes de que lleguen a producción y poder bloquear el deploy si algo está roto.

## Descripción

Hoy no existen tests automatizados en el proyecto. Cualquier push a producción podría romper silenciosamente lógica crítica como los cálculos financieros de operaciones, las validaciones de los endpoints, o el comportamiento del middleware de autenticación. Esta historia cubre la creación de la infraestructura de testing (configuración de Vitest + GitHub Actions) y los primeros unit tests sobre las funciones puras del sistema. No incluye tests de integración ni E2E.

Para poder testear la lógica de cálculos financieros que hoy vive inline dentro de las API routes, es necesario extraerla primero a funciones puras en un módulo separado. Lo mismo aplica para los helpers del middleware, que actualmente no están exportados.

## Criterios de aceptación

- [ ] Existe un archivo `vitest.config.ts` configurado correctamente en el proyecto
- [ ] Existe un workflow de GitHub Actions (`.github/workflows/tests.yml`) que se dispara en cada push y PR hacia `main`
- [ ] Si algún test falla, el workflow queda en estado de error y bloquea el merge/push
- [ ] Existe el módulo `src/lib/calculations.ts` con las funciones `calcularIngresosNetos` y `calcularComision` extraídas de las API routes
- [ ] Los helpers `isPublicPath`, `extractClienteIdFromPath` y `buildLoginUrl` del middleware están exportados y son testeables
- [ ] Existen tests unitarios para `calcularIngresosNetos` cubriendo: caso normal, sin gastos asociados, y comisión cero cuando el precio de venta es cero
- [ ] Existen tests unitarios para `calcularComision` cubriendo: caso normal, precio de venta cero (no divide por cero), e ingresos netos negativos
- [ ] Existen tests unitarios para `isValidOperationTypeName` cubriendo: tipos válidos, el id del tipo (no el nombre), y string vacío
- [ ] Existen tests unitarios para `getOperationTypeByName` cubriendo: nombre existente y nombre inexistente
- [ ] Existen tests unitarios para los helpers del middleware cubriendo: rutas públicas, rutas privadas, extracción de clienteId de la URL, y construcción de URL de login con y sin parámetro `expired`
- [ ] La suite completa corre en menos de 10 segundos

## Flujos

### Flujo principal (escenario de éxito)

1. El equipo hace push a `main` o abre un PR hacia `main`
2. GitHub Actions detecta el evento y dispara el workflow `tests.yml`
3. El workflow instala dependencias y ejecuta `vitest run`
4. Todos los tests pasan
5. El pipeline queda en estado verde
6. El merge/push se permite proceder

### Flujo alternativo 1 — Tests fallan

1. El equipo hace push a `main` o abre un PR hacia `main`
2. GitHub Actions dispara el workflow
3. Uno o más tests fallan (por ejemplo: se modificó la fórmula de comisión y el resultado ya no coincide con el esperado)
4. El pipeline queda en estado rojo con el detalle del test fallido
5. El merge/push queda bloqueado hasta que se corrija el código o el test

### Flujo alternativo 2 — Micro-refactor previo necesario

1. Antes de escribir los tests de cálculos financieros, se extrae la lógica inline de las API routes hacia `src/lib/calculations.ts`
2. Antes de escribir los tests del middleware, se exportan los helpers `isPublicPath`, `extractClienteIdFromPath` y `buildLoginUrl`
3. Se verifica que las API routes sigan funcionando igual usando las funciones extraídas
4. Recién entonces se escriben los tests sobre esos módulos

## Notas técnicas

- **Framework de tests:** Vitest (ya instalado como devDependency)
- **Archivos a crear:**
  - `vitest.config.ts` — configuración de Vitest
  - `src/lib/calculations.ts` — funciones puras de cálculos financieros (extraídas de las API routes)
  - `src/__tests__/calculations.test.ts` — tests de cálculos financieros
  - `src/__tests__/operation-types.test.ts` — tests de validación de tipos de operación
  - `src/__tests__/middleware.test.ts` — tests de helpers del middleware
  - `.github/workflows/tests.yml` — workflow de GitHub Actions
- **Archivos a modificar:**
  - `src/app/api/operations/route.ts` — usar funciones de `calculations.ts` en lugar de lógica inline
  - `src/app/api/operations/[id]/route.ts` — ídem
  - `src/middleware.ts` — exportar los helpers de ruta
- **No requiere base de datos:** todos los tests de esta HU son sobre funciones puras que no tocan Prisma ni la sesión
